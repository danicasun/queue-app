High-Level Summary
- Next.js App Router with /app as the authenticated area and /login for auth entry; root landing at /.
- Server Actions handle nearly all data access (lib/actions/*), with client components calling them directly.
- Supabase client creation is split between lib/supabase/server.ts and direct @supabase/ssr usage in app/auth/callback/route.ts, creating inconsistency.
- Middleware protects /app routes by user presence; if env vars are missing, protection silently disables.
- RLS is enabled with policies in supabase/schema.sql, but some tables (e.g., queue_items, folders) appear unused in app code.
- Several client pages (/app/settings, /app/organize) are fully client-rendered with placeholder state; no server data.
- Validation is ad hoc and duplicated in lib/validation.ts; no schema library like Zod.
- Legacy and reference folders (legacy-pages/, ui-reference/, Vite-era helpers) likely add confusion and dead code.
- Caching is limited to revalidatePath, with no explicit fetch caching or tagging; client state can become stale.

System Mental Model

What the app does
Queue is a personal learning queue app. Users log in via magic link, maintain a list of topics, attach resources, and optionally share topics to a Discover feed. Basic flows exist for topic CRUD, resource CRUD, and comments on shared topics.

Core user flows
- Auth: Magic link sign-in (/login -> /auth/callback), then redirect to /app.
- Queue: View topics, add topic, view detail, update title/note/visibility, add/delete resources.
- Discover: View shared topics from others, save a shared topic into your own queue, and comment on shared topics.
- Organize/Settings: Placeholder client UI with local-only state.

Data flow diagram (written)
- UI (client components) -> Server Actions (lib/actions/*.ts) -> Supabase (PostgREST via @supabase/ssr).
- For some read flows (e.g., comments), client components invoke server actions from useEffect, which triggers a network call back to the server action -> Supabase -> returns data -> client state update.

Structural Issues / Red Flags (with file paths)

Critical
- Route protection depends on env vars being set. If NEXT_PUBLIC_SUPABASE_URL or NEXT_PUBLIC_SUPABASE_ANON_KEY is missing, middleware.ts becomes a no-op, leaving /app/* unprotected. (middleware.ts)
- Inconsistent Supabase client creation. Direct @supabase/ssr usage in app/auth/callback/route.ts bypasses lib/supabase/server.ts (different cookie handling assumptions). (app/auth/callback/route.ts, lib/supabase/server.ts)

High
- Client-triggered server actions for reads. Comment loading happens in a client useEffect, which means data is always fetched client-side instead of server-rendered. This can cause waterfalls and inconsistent caching. (components/queue/CommentSheet.tsx, lib/actions/discover.ts)
- Potential stale client state after server mutations. UI uses local useState updates with optimistic assumptions; server revalidation does not trigger refetch in the client, so cross-tab or concurrent edits can desync. (app/app/QueueClient.tsx, app/app/topic/[id]/TopicDetailClient.tsx)

Medium
- Excessive client components. app/app/organize/page.tsx and app/app/settings/page.tsx are client-only despite being mostly static. This increases bundle size and reduces SSR benefits.
- Validation inconsistency. lib/validation.ts defines two overlapping validation APIs; only a subset is used. This will confuse future refactors.
- Legacy and reference code in repo. legacy-pages/ and ui-reference/ appear unused; lib/app-params.js references import.meta.env (Vite-era), and lib/query-client.js references React Query without a dependency in package.json.

Low
- Hard-coded placeholders. Organize and Settings pages show placeholder data (e.g., fake tags and profile), which may mask missing backend wiring.
- No explicit error boundaries. App Router lacks error.tsx, so runtime failures fall back to global errors without route-level recovery.

Supabase Architecture

Client creation
- Server: lib/supabase/server.ts (cookies from next/headers)
- Browser: lib/supabase/client.ts (unused)
- Route handler callback uses direct @supabase/ssr createServerClient (inconsistent with local wrapper)

Auth flow
- /login triggers signInWithEmail server action, redirect to /auth/callback
- app/auth/callback/route.ts exchanges code -> session
- middleware.ts checks session on /app/* and /login to redirect

Database
- Tables used: topics, resources, tags, topic_tags, comments
- Tables unused: folders, queue_items, likely profiles

RLS
- Enabled for all main tables
- Shared select policies allow any authenticated user to see non-private topics/resources/comments
- Current app relies on visibility and user-scoped policies for access control

Storage
- No evidence of Supabase Storage usage in code

Service role
- No service role key found in code; good

Cleanup Plan (overview only, with hypotheses)

Phase 1: Consolidate Supabase client creation
Done when app/auth/callback/route.ts and all server code use one createServerClient factory and no direct @supabase/ssr usage remains.
Hypothesis: A single factory reduces auth cookie bugs and keeps server/client boundaries consistent.

Phase 2: Remove or quarantine legacy/reference code
Done when legacy-pages/, ui-reference/, lib/app-params.js, and lib/query-client.js are either removed or moved under an explicit archive/ folder with README context.
Hypothesis: Removing unused code will cut onboarding time and prevent accidental imports.

Phase 3: Normalize validation and error handling
Done when all server actions use one validation API (e.g., lib/validation.ts simplified or replaced with Zod).
Hypothesis: Consistent validation reduces duplicate logic and makes errors predictable.

Phase 4: Tighten client/server boundaries
Done when static pages render as server components unless interactivity is required, and comment loading is server-rendered or uses a dedicated API with proper cache control.
Hypothesis: Reducing unnecessary client components improves performance and simplifies data flow.

Phase 5: Define data access layer (DAL)
Done when Supabase queries are centralized (e.g., lib/data-access/) and server actions call the DAL instead of inline queries.
Hypothesis: Centralized queries reduce duplication and make RLS assumptions explicit.

Phase 6: Align schema with product
Done when unused tables are removed or used, and product features (Organize/Settings) are backed by DB or explicitly marked as mock.
Hypothesis: Schema-app alignment prevents drift and keeps future changes safer.

Optional: Golden Path Recommendation

Auth + RLS best practice (for this repo)
- Use createServerClient (with @supabase/ssr) everywhere on the server.
- Use createBrowserClient only in true client-only features.
- Read auth via supabase.auth.getUser() in server actions and middleware.
- Keep NEXT_PUBLIC_SUPABASE_ANON_KEY for client, reserve service role for server-only operations if ever needed.
Hypothesis: Following SSR + RLS with anon key keeps access consistent and minimizes accidental privilege escalation.

Ideal placement
- supabaseClient.ts: lib/supabase/server.ts and lib/supabase/client.ts (keep as the only entry points).
- Server actions: lib/actions/* (but backed by a DAL in lib/data-access/).
- Zod schemas: lib/validation/* or lib/schemas/*.
- Domain/data-access layer: lib/data-access/* (one file per aggregate: topics.ts, resources.ts).
- Shared types: lib/types/* or types/* at root.
Hypothesis: A dedicated DAL + schemas keeps data contracts stable and reduces coupling between UI and database details.
